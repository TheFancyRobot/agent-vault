Refine every step in a phase until the work is clear enough for a first-day junior developer.

Usage: /vault:refine <phase-id>

Accept a required phase id such as `PHASE-01`.

Workflow:

1. Resolve the phase, then understand the whole phase before touching individual steps.
   - Find the exact phase note under `.agent-vault/02_Phases/` by matching `phase_id:`.
   - Load focused context with `vault_traverse`: the phase, all linked step notes, related architecture, related bugs, related decisions, and recent sessions.
   - Read the full phase objective, scope, non-goals, dependencies, acceptance criteria, and notes.
   - Research the codebase paths referenced across the whole phase so you understand how the steps fit together before asking any questions.

2. Use tactical subagent research to build phase-wide understanding first.
   - Launch independent research in parallel when useful, for example:
     - `explore` for code-path discovery and existing patterns
     - `general` for likely missing requirements, integration risks, and validation gaps
     - `security-reviewer` or `general` for security, privacy, and misuse concerns when the phase touches sensitive behavior
   - Synthesize the results into a phase-level map of shared workflows, constraints, code paths, edge cases, and unknowns.
   - Do not ask the user questions yet unless a single global ambiguity blocks understanding of the entire phase.

3. Generate and apply a mandatory readiness checklist for every step.
   - The checklist is generated by you from the phase template, step template, related vault context, and codebase research. Do not wait for the user to supply one.
   - Use the same checklist for every step in the phase so quality stays consistent.
   - For each step, explicitly assess whether a junior developer could start and finish it safely using only the note, linked context, and obvious codebase conventions.
   - The readiness checklist must cover, at minimum:
     - exact outcome and success condition
     - why the step matters to the phase
     - prerequisites, setup state, and dependencies
     - concrete starting files, directories, packages, commands, and tests
     - required reading completeness
     - implementation constraints and non-goals
     - validation commands, manual checks, and acceptance criteria mapping
     - edge cases, failure modes, and recovery expectations
     - security considerations or explicit not-applicable judgment
     - performance considerations or explicit not-applicable judgment
     - integration touchpoints and downstream effects
     - blockers, unresolved decisions, and handoff expectations
     - junior-developer readiness verdict: pass or fail
   - A step does not pass refinement until every checklist item is either concretely answered or explicitly marked not applicable with a short reason.

4. Ask clarifying questions only after doing the research and initial checklist pass, and make them high leverage.
   - Ask the user targeted clarifying questions that unblock the most ambiguity first.
   - Prefer one short batch per step, or one global batch plus one step-specific batch when that is more efficient.
   - Keep each batch small and concrete: usually 2-4 questions maximum.
   - Ask once for anything that affects multiple steps instead of repeating the same question later.
   - Prefer questions with crisp options, concrete scenarios, defaults, or tradeoffs instead of open-ended prompts.
   - Prioritize checklist failures that would cause rework, hidden assumptions, unsafe implementation, or validation confusion.

5. Turn the answers into stronger step requirements immediately and rerun the checklist.
   - Update the phase note and each step note conservatively so the new requirements live with the work, not only in chat history.
   - Use `vault_mutate` for frontmatter updates and append-only note sections.
   - When a planning section itself needs to become more specific, make careful targeted markdown edits rather than broad rewrites.
   - Strengthen the step until it includes explicit guidance a junior engineer can follow:
     - exact outcome
     - concrete file or directory starting points
     - required reading links
     - implementation constraints and non-goals
     - validation commands or manual checks
     - edge cases and error handling expectations
     - security and performance requirements when relevant
   - If new durable defects or design choices are discovered, create linked notes with `vault_create`.
   - After each update, re-evaluate the step against the full readiness checklist and keep iterating until the step passes.

6. Keep iterating until the phase is execution-ready.
   - After refining one step, move to the next and revisit earlier steps if later answers change the shared understanding.
   - Do not stop at "good enough for a senior engineer." The bar is: a first-day junior developer can execute the step without hidden context or guesswork.
   - If something still cannot be clarified, record it explicitly as a blocker or unresolved decision instead of leaving it implicit.
   - If the user declines to answer a needed question, mark the affected checklist items as blocked and say exactly why the step is not yet junior-ready.

7. Close out the refinement pass.
   - Leave the phase and its steps more explicit, more actionable, and more testable than before.
   - Summarize the checklist status for each step, the remaining open questions, if any, and identify which steps are now ready for execution.
   - If an active session exists for this conversation, append a summary of the refinement work to the session's `Execution Log` and update `Follow-Up Work` with the recommended next action (typically execution of the refined phase).
   - Finish with `vault_refresh` so home notes stay current, then run `vault_validate` to confirm the vault still passes integrity checks.

Refinement is complete only when each step passes the generated readiness checklist and has enough concrete context, constraints, validation guidance, and clarified requirements that a new junior developer could pick it up and execute safely.

Examples:
- /vault:refine PHASE-01
- /vault:refine PHASE-07
